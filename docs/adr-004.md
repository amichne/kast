# ADR-004: Capability negotiation over backend polymorphism

**Status**: Accepted

**Context**: The two backends will never have identical feature sets. The standalone backend will always lag on mutations. Hard-coding "if standalone then skip" logic in every consumer is brittle and doesn't scale.

**Decision**: The `/capabilities` endpoint returns a structured declaration of what this backend instance supports. Clients check capabilities before calling mutation operations. `MutationResult.Unsupported` is a first-class result type for operations that slip through.

**Consequences**:
Positive: New capabilities are additive — add an enum value, implement in one or both backends, consumers discover it automatically.
Negative: Clients need to handle the capability check. Acceptable given LLM agents already handle conditional branching.
