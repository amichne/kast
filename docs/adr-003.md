# ADR-003: Mutations modeled as text edits, not PSI operations

**Status**: Accepted

**Context**: The two backends have fundamentally different mutation primitives. IntelliJ has `WriteAction` + PSI manipulation. Standalone has file I/O. The RPC contract must abstract over both without leaking backend details.

**Decision**: All mutations produce `List<TextEdit>` (file path + offset range + replacement text). `dryRun: true` by default — the mutation is computed and returned but not applied. The client decides whether to call `applyEdits`.

This is also the correct representation for LLM agents, which reason about text diffs, not PSI trees.

**Alternatives Considered**:
*PSI-level mutation descriptions*: Richer (e.g., "rename this KtNamedFunction"), but leaks IntelliJ internals and is meaningless to the standalone backend.

*Unified diff format*: Familiar to developers, but harder to apply programmatically and loses offset precision.

**Consequences**:
Positive: Backend-agnostic. LLM-friendly. Composable (agent can merge edits from multiple operations).
Negative: Offset-based edits are fragile if the file changes between compute and apply. Mitigated by the dry-run-then-apply two-step and by embedding file content hashes in future versions.
