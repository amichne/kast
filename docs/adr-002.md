### ADR-002: HTTP + JSON over LSP for the wire protocol

**Status**: Accepted

**Context**: The consumers are LLM tool-calling agents, not editors. LSP assumes a persistent client with open-document state, capability negotiation, and incremental text sync. None of that applies here. Each request is stateless ("resolve the symbol at offset 142 in Foo.kt").

**Decision**: Use plain HTTP + JSON. Ktor embedded server, kotlinx.serialization for JSON. One route per operation.

**Alternatives Considered**:
*LSP (JSON-RPC over stdio/socket)*: Full protocol implementation for ~5% of the protocol surface we'd actually use. `textDocument/didOpen` lifecycle management, `workspace/configuration` negotiation, progressive document sync — all dead code for batch-style tool calls.

*gRPC*: Stronger typing and streaming, but adds protobuf compilation step, proto file management, and gRPC runtime weight. Overkill for a localhost tool with less than 10 operations.

**Consequences**:
Positive: Trivial to test with `curl`. No protocol library dependencies. Easy to add routes.
Negative: No built-in streaming for large results (if call hierarchy is deep). Solvable with pagination or SSE if needed later.
