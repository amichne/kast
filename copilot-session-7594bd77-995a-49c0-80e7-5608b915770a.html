## Good mental model

Kotter is not "manual ANSI with nicer names." It is a small reactive terminal framework:

1. `session` owns shared state and lifecycles.
2. `section` defines the active render area.
3. `run` handles async work, listeners, and timers.
4. live state drives rerenders.
5. `aside` writes history above the active area.
6. `offscreen` lets you compose and measure before emitting.

If you keep that model in mind, most API choices become obvious.

# Composition and layout

Use this file when the task is about arranging terminal content, not just styling it.

## Think in two areas

Kotter output usually has:

1. **Static history** above
2. **Active rendering area** at the bottom

Use separate finished sections for static instructions. Use `aside` when you need to append history while an active section is still rerendering.

## `section` vs `aside`

### `section`

Use `section` for the active, rerendered area:

```kotlin
section {
    textLine("Searching...")
}.run { ... }
```

### `aside`

Use `aside` from inside a `run` block when long-running work emits durable history:

```kotlin
aside { textLine("Compiled Foo.kt") }
```

This is a strong fit for:

- compilers
- test runners
- chat transcripts
- logs emitted while a spinner or status panel stays active

## `offscreen`

Use `offscreen` when you need to render first, then measure or replay the result.

It gives you:

- `lineLengths`
- `numLines`
- `createRenderer()`

This is the core primitive behind higher-level composition helpers.

### Common `offscreen` use cases

1. Measure content width before drawing a frame.
2. Right-align or center a previously rendered block.
3. Replay content row-by-row while wrapping it with other characters.
4. Build your own decoration helpers.

### Important property

An offscreen buffer inherits parent style initially, but local style changes inside the buffer stay local to the buffer.

## `bordered`

Use `bordered(...)` when a screen area needs to read like a panel:

```kotlin
bordered(
    borderCharacters = BorderCharacters.CURVED,
    paddingLeftRight = 1,
    paddingTopBottom = 1,
) {
    textLine("Panel title")
    textLine("Panel body")
}
```

Available border styles:

- `BorderCharacters.ASCII`
- `BorderCharacters.BOX_THIN`
- `BorderCharacters.BOX_DOUBLE`
- `BorderCharacters.CURVED`

Use ASCII when portability matters more than polish.

## `grid`

Use `grid(...)` when content is truly tabular or when you want side-by-side panels.

### Column types

- `fixed(width)`
- `fit()`
- `star(ratio = ...)`

Use `targetWidth` when star-sized columns need real room distribution.

### Column builder

```kotlin
grid(
    Cols {
        fit(maxWidth = 24)
        fixed(8, justification = Justification.CENTER)
        star(minWidth = 10)
    },
    targetWidth = 80,
) { ... }
```

### Grid features worth remembering

- `colSpan`
- `rowSpan`
- `nextEmptyCellRow`
- `nextEmptyCellCol`
- per-cell `justification`
- alternate border styles via `GridCharacters`

### Border styles

- `GridCharacters.ASCII`
- `GridCharacters.BOX_THIN`
- `GridCharacters.BOX_DOUBLE`
- `GridCharacters.CURVED`
- `GridCharacters.INVISIBLE`

`INVISIBLE` is useful when you want table-like flow without visible walls.

### Grid guidance

- Bound cell content before it reaches the grid.
- Keep preview columns short.
- Use grids for comparisons, summaries, inventories, or dashboards.
- Do not dump full source lines or huge payloads into narrow cells.

## `justified`

Use `justified(...)` when you need alignment over a rendered block:

- `Justification.LEFT`
- `Justification.CENTER`
- `Justification.RIGHT`

This is useful for:

- centered banners
- aligned labels
- padding content inside custom decorated layouts

`padRight = false` can reduce trailing whitespace when alignment is all you care about.

## `shiftRight`

Use `shiftRight(amount)` when content should be horizontally offset without building a grid just for that:

```kotlin
shiftRight(20) {
    textLine("Indented content")
}
```

It is especially useful for:

- splash screens
- centered-ish art
- nested focus blocks

## Composition recipes

### Boxed status panel

Use `bordered` around a compact render model.

### Diff or comparison panel

Use `grid` with two or three columns, then a summary line below.

### Measured custom layout

Use `offscreen` to measure text width, then replay it with your own padding or framing.

### Live progress with durable history

Use a status `section` plus `aside` calls from the run block.

# Input and interaction

Use this file when the terminal UI accepts input, reacts to keys, or supports navigation.

## Typed input

The basic pattern is:

```kotlin
section {
    text("Name: ")
    input()
}.runUntilInputEntered {
    onInputEntered { name = input }
}
```

`input()`:

- rerenders as text changes
- manages cursor display
- handles arrow keys plus home/end

## Input validation

### On each keystroke

Use `onInputChanged` when invalid characters should be rejected immediately:

```kotlin
onInputChanged {
    if (input.any { !it.isDigit() }) rejectInput()
}
```

### On enter

Use `onInputEntered` when the full value should be validated at commit time:

```kotlin
onInputEntered {
    if (input.isBlank()) rejectInput()
    else accepted = input
}
```

## Completions

Kotter supports custom `InputCompleter` implementations, plus the built-in:

```kotlin
input(Completions("yes", "no"))
```

Use this for:

- prompts with a small known vocabulary
- command palettes
- confirmations

Remember that completion order matters.

## `viewMap` and `customFormat`

### `viewMap`

Use this for visual-only transformations like password masking:

```kotlin
input(viewMap = { '*' })
```

The underlying input remains unchanged.

### `customFormat`

Use this for character-level styling:

```kotlin
input(customFormat = {
    if (ch.isDigit()) green() else red()
})
```

This is good for:

- showing valid vs invalid characters
- highlighting delimiters
- basic inline syntax cues

## Multiple inputs in one section

This is supported, but there are two rules:

1. the helper must live on `MainRenderScope`
2. every input needs a unique `id`

Pattern:

```kotlin
fun MainRenderScope.channelInput(line: Int, label: String) {
    text("$label: ")
    input(id = line, isActive = selectedLine == line)
    textLine()
}
```

Keep focus state outside the helper.

## Multiline input

Use `multilineInput()` when ENTER should insert a newline instead of submitting the field.

Important behavior:

- finish with CTRL-D / EOF
- arrow/home/end/page keys navigate within the text
- content after the multiline input starts on a later line

This is a good fit for:

- commit messages
- chat prompts
- note entry

## Key handling

Use `onKeyPressed` for semantic key events:

```kotlin
onKeyPressed {
    when (key) {
        Keys.UP -> ...
        Keys.DOWN -> ...
        Keys.ENTER -> ...
        Keys.ESC -> ...
    }
}
```

Useful keys include:

- arrows
- `ENTER`
- `ESC`
- `TAB`
- `HOME`, `END`
- `PAGE_UP`, `PAGE_DOWN`
- `SPACE`
- `EOF`

For literal typed characters, you may receive `CharKey`.

## Convenience runners

Use:

- `runUntilInputEntered`
- `runUntilKeyPressed(Keys.Q)`
- `runUntilSignal`

These express intent better than hand-written loops in common cases.

## Programmatic input control

Kotter exposes helpers that are easy to forget but useful:

- `getInput(id)`
- `setInput(text, cursorIndex, id)`
- `enterInput()`
- `clearInput()`
- `sendKeys(...)`

These are useful for:

- seeding a field with previous content
- carrying a draft into the next interaction step
- injecting keys from inside a running section

## Navigation patterns

### 1. Arrow-key list

Use a live cursor index plus `onKeyPressed`:

```kotlin
var cursorIndex by liveVarOf(0)

section {
    items.forEachIndexed { i, item ->
        text(if (i == cursorIndex) '>' else ' ')
        text(' ')
        textLine(item.label)
    }
}.runUntilSignal {
    onKeyPressed {
        when (key) {
            Keys.UP -> cursorIndex = (cursorIndex + items.size - 1) % items.size
            Keys.DOWN -> cursorIndex = (cursorIndex + 1) % items.size
            Keys.ENTER -> signal()
        }
    }
}
```

### 2. Viewport navigator

For richer UIs, keep:

1. a tiny state object
2. a pure `advance(key)` function
3. a pure `focus(...)` or `view(...)` function
4. a renderer that only consumes focused state

This is the best pattern for file navigation, ripples, tab sets, or paged detail panels.

# Reactivity, lifecycle, and long-running behavior

Use this file when the terminal UI needs to stay alive, react to state, or coordinate work over time.

## Scopes

Kotter is easiest to use when you keep the scopes straight.

### `Session`

Owns:

- shared state
- `liveVarOf`
- `liveListOf`, `liveMapOf`, `liveSetOf`
- section composition

### `RenderScope`

Owns drawing:

- text
- styling
- layout helpers

### `RunScope`

Owns behavior:

- listeners
- timers
- signals
- shutdown hooks
- asynchronous orchestration

## `run` and rerendering

The `section { ... }` block can render more than once. A `run { ... }` block can:

- change state
- wait for work
- trigger rerenders

Manual `rerender()` works, but prefer live state when possible.

## `LiveVar`

Use:

```kotlin
var result by liveVarOf<Int?>(null)
```

When you update a `LiveVar`, Kotter automatically requests another render.

Choose `LiveVar` when:

- one value drives visible output
- state changes happen over time
- you want to avoid remembering explicit `rerender()` calls

## Live collections

Kotter also ships:

- `liveListOf`
- `liveMapOf`
- `liveSetOf`

They rerender when mutated.

### Locking guidance

If you are reading or writing several properties together from a run block, use:

- `withReadLock { ... }`
- `withWriteLock { ... }`

This matters most when:

- trimming a rolling window
- updating multiple related collection entries
- reading several values that must stay consistent

## Signals

Kotter provides:

- `signal()`
- `waitForSignal()`
- `runUntilSignal { ... }`

Use them when the run block should stay alive until some event fires.

This is usually cleaner than hand-rolling a latch or deferred.

## Timers

Use:

```kotlin
addTimer(duration, repeat = true) { ... }
```

Timer callbacks can:

- flip live state
- stop repetition by setting `repeat = false`
- chain follow-up timers
- signal when done

Kotter also supports:

- `runFor`
- `runForAtLeast`
- `runForAtMost`

These are useful for bounded waits or small time-boxed interactions.

## `onFinishing`

Use `onFinishing` for cleanup when timers or toggled state may leave the final screen in the wrong condition.

Typical cases:

- blinking or inverted state
- temporary warning banners
- transient selections that should normalize on exit

## Animations

Kotter supports:

- `textAnimOf`
- `renderAnimOf`
- templates
- one-shot animations (`looping = false`)

Use `textAnimOf` for simple string-frame animations.

Use `renderAnimOf` when frames need styling, layout, or arbitrary render logic.

Useful properties:

- `totalDuration`
- `currFrame`

Reset `currFrame = 0` to restart a one-shot animation.

## Shutdown hooks

Use `addShutdownHook { ... }` when CTRL-C should trigger a short cleanup action or last visible warning.

There are session- and run-scoped variants.

Rules:

- keep them short
- do not rely on them for correctness
- treat the extra render opportunity as best-effort only

## Thread model

Kotter is intentionally centered on one active rendering area at a time.

- sections render sequentially on one render thread
- the run block executes on the calling thread
- trying to run competing sections concurrently is the wrong shape

Design your UI as a sequence of sections, not as multiple active terminals fighting each other.

# Kast

IDE-grade Kotlin analysis and refactoring, exposed as a single executable
that speaks JSON in / JSON out. A companion hook guarantees
`KAST_CLI_PATH` points at the binary before this skill runs, so every
command in this document invokes `"$KAST_CLI_PATH"` directly.

## Shape of every call

    "$KAST_CLI_PATH" skill <command> <request>

`<request>` is **exactly one argument** and may be either:

1. an inline JSON object literal (single-quoted in the shell), or
2. an absolute path to a `.json` file containing the same object.

The response is a single JSON document on stdout. Anything on stderr is
lifecycle chatter from the daemon and is safe to ignore when `ok=true`.

Every successful response carries:

- `type` — discriminator tag (for example `RESOLVE_SUCCESS`,
  `REFERENCES_FAILURE`, `WRITE_AND_VALIDATE_SUCCESS`).
- `ok` — boolean. Treat `false` as a failed transaction: stop the
  current plan, read `log_file`, fix the cause, rerun.
- `query` — a normalized echo of the inputs. Useful when chaining
  commands so you can keep using the resolved `file_path` / `offset`.
- `log_file` — absolute path to the per-request daemon log.

## Casing is asymmetric — don't fight it

- **Request input** uses camelCase (`workspaceRoot`, `filePath`,
  `newName`, `includeDeclaration`, `maxTotalCalls`).
- **Response output** uses snake_case (`workspace_root`, `file_path`,
  `new_name`, `include_declaration`, `max_total_calls`).

Both shapes come straight from the Kotlin contract types. Don't
translate them; the daemon won't accept camelCase in responses or
snake_case in requests. The full schema lives in
`references/wrapper-openapi.yaml` — consult it before guessing.

## Workspace root

Every request targets one workspace. Resolution order:

1. `workspaceRoot` inside the request body
2. `KAST_WORKSPACE_ROOT` environment variable

No implicit "use git to find the root" fallback exists. If neither is
set the command fails with a `SKILL_VALIDATION` error. Pick one and
stick with it for the session.

| Check error/warning state for files | `skill diagnostics` |
| Rename a symbol across the workspace | `skill rename` |
| Gather implementation context for a target | `skill scaffold` |
| Create / insert / replace, then validate | `skill write-and-validate` |
| List modules and source files | `skill workspace-files` |

Two of these — `rename` and `write-and-validate` — are polymorphic and
require a `type` discriminator in the request body (see below).

## How to think about the workflow

Treat every non-trivial task as three phases:

1. **Navigate.** Turn the user's name for a thing into a concrete
   position with `resolve`, `workspace-files`, or `scaffold`. These are
   cheap and idempotent.
2. **Act.** Do the one mutation that matters — `rename`,
   `write-and-validate`, or a chain of reads with `references` /
   `callers` to decide what to write.
3. **Validate.** `diagnostics` on the files you touched, plus the
   diagnostics summary already embedded in mutation responses. Don't
   report success until both agree.

Skipping navigate → act produces wrong edits. Skipping act → validate
produces plausible-looking edits that break the build.

## Reference recipes

### Resolve a symbol

    "$KAST_CLI_PATH" skill resolve '{
      "symbol":"AnalysisServer",
      "fileHint":"analysis-server/src/main/kotlin/io/github/amichne/kast/server/AnalysisServer.kt"
    }'

Narrow ambiguous matches with:

- `kind` — one of `class`, `interface`, `object`, `function`, `property`.
- `containingType` — fully qualified name of the enclosing declaration.
- `fileHint` — absolute or workspace-relative path. When a symbol exists
  in several modules, this is usually the fastest disambiguator.

### Find references

    "$KAST_CLI_PATH" skill references '{
      "symbol":"AnalysisServer",
      "includeDeclaration":true
    }'

`includeDeclaration` defaults to `true`; set it to `false` when you want
only call-sites, not the declaration itself.

### Walk the call hierarchy

    "$KAST_CLI_PATH" skill callers '{
      "symbol":"process",
      "direction":"incoming",
      "depth":3,
      "maxTotalCalls":256,
      "maxChildrenPerNode":64
    }'

- `direction` is `incoming` (who calls me) or `outgoing` (who I call).
- `depth` caps recursion. Default 2; increase deliberately — the graph
  grows fast.
- `maxTotalCalls` / `maxChildrenPerNode` bound the traversal so it
  stays responsive. Defaults are 256 and 64.

### Run diagnostics

    "$KAST_CLI_PATH" skill diagnostics '{
      "filePaths":[
        "/abs/path/File.kt",
        "/abs/path/Other.kt"
      ]
    }'

Paths must be absolute. The response's `clean` field is your quick
green-light; `errors` contains anything that would prevent a build.

### Rename a symbol — by name

    "$KAST_CLI_PATH" skill rename '{
      "type":"RENAME_BY_SYMBOL_REQUEST",
      "symbol":"OldName",
      "newName":"NewName"
    }'

### Rename a symbol — by explicit position

Use this when the name is ambiguous (e.g. an overload or a property
that shares a name with a class).

    "$KAST_CLI_PATH" skill rename '{
      "type":"RENAME_BY_OFFSET_REQUEST",
      "filePath":"/abs/path/File.kt",
      "offset":1234,
      "newName":"NewName"
    }'

The typical pattern is `resolve` → read `file_path` and `offset` from
the response → `rename` with `RENAME_BY_OFFSET_REQUEST`. This sidesteps
any naming collisions the workspace might contain.

Either variant returns an `ApplyEditsResult`, a list of
`affected_files`, and a `diagnostics` summary. `ok=true` means the
workspace compiles cleanly after the rename.

### Scaffold implementation context

Before writing a new implementation, pull everything you need in one
request:

    "$KAST_CLI_PATH" skill scaffold '{
      "targetFile":"/abs/path/Interface.kt",
      "targetSymbol":"MyInterface",
      "mode":"implement"
    }'

You get the file outline, the current file content, the resolved
target symbol, its references and type hierarchy, and a
`semantic_insertion` offset tailored to `mode`:

| mode | Where insertion points to |
| --- | --- |
| `implement` | end of the class body |
| `replace` | start of the class body |
| `consolidate` | bottom of the file |
| `extract` | after the imports |

### Write and validate — create a file

    "$KAST_CLI_PATH" skill write-and-validate '{
      "type":"CREATE_FILE_REQUEST",
      "filePath":"/abs/path/NewImpl.kt",
      "content":"package foo\n\nclass NewImpl"
    }'

### Write and validate — insert at an offset

    "$KAST_CLI_PATH" skill write-and-validate '{
      "type":"INSERT_AT_OFFSET_REQUEST",
      "filePath":"/abs/path/File.kt",
      "offset":42,
      "content":"override fun x() = y()\n"
    }'

### Write and validate — replace a range

    "$KAST_CLI_PATH" skill write-and-validate '{
      "type":"REPLACE_RANGE_REQUEST",
      "filePath":"/abs/path/File.kt",
      "startOffset":120,
      "endOffset":240,
      "content":"…"
    }'

For all three variants you may pass `contentFile` (absolute path) in
place of `content`. Prefer that when the payload is large, contains
newlines, or is built from another tool — it avoids shell-quoting
pitfalls and keeps the command line readable.

Every write-and-validate response has already run import optimization
and diagnostics for you; `ok` is true only when both came back clean.

### List workspace files

    "$KAST_CLI_PATH" skill workspace-files '{"includeFiles":true}'

Without `includeFiles` you get modules, source roots, and file counts —
the cheap version. Add `moduleName` to narrow to one module.

fun RenderScope.renderRippleAct(root: CallerNode, depth: Int) {
    renderActHeader(

        subtitle = "",
    )

    textLine()

    val allNodes = collectNodes(root)
    val uniqueModules = allNodes.map { it.module }.toSet()
    val moduleWord = if (uniqueModules.size == 1) "module" else "modules"
    val symbolWord = if (allNodes.size == 1) "symbol" else "symbols"

    text("  ")
    green(isBright = true) { text("${uniqueModules.size} $moduleWord") }
    text(". ")
    green(isBright = true) { text("${allNodes.size} $symbolWord") }
    textLine(" reachable in $depth hops.")

    black(isBright = true) { textLine("  Every edge is a compiler-verified call site.") }
    textLine()
    black(isBright = true) {
        textLine("  kast demo --symbol <fqn> --depth ${depth + 1}")
    }
}

private fun RenderScope.renderTreeNode(
    node: CallerNode,
    prefix: String,
    isLast: Boolean,
    isRoot: Boolean,
    currentDepth: Int,
) {
    if (isRoot) {
        text(prefix)
        cyan(isBright = true) { text(node.symbolName) }
    } else {
        text(prefix)
        val branch = if (isLast) "└── " else "├── "
        text(branch)
        when {
            currentDepth <= 1 -> yellow(isBright = true) { text(node.symbolName) }
            else -> text(node.symbolName)
        }
    }

    // Module label
    text("  ")
    color(ModulePalette.colorFor(node.module)) {
        text("[${node.module}]")
    }
    textLine()

    // Render children
    val childPrefix = if (isRoot) {
        prefix
    } else {
        prefix + if (isLast) "    " else "│   "
    }

    node.children.forEachIndexed { index, child ->
        renderTreeNode(
            node = child,
            prefix = childPrefix,
            isLast = index == node.children.lastIndex,
            isRoot = false,
            currentDepth = currentDepth + 1,
        )
    }
}

private fun collectNodes(node: CallerNode): List<CallerNode> {
    return listOf(node) + node.children.flatMap { collectNodes(it) }
}

# kast — Terminal UI Design Specification
### Feature Ticket: Interactive Compiler Semantic Demo

---

## Purpose

This document specifies the visual and behavioral design of the `kast` interactive terminal UI. It is
implementation-agnostic at the element level, with feasibility notes scoped to Kotter (Kotlin terminal
rendering library) called out explicitly per element. It is intended as the planning surface for an
agent tasked with implementation — outlining *what* each element is, *why* it earns its complexity,
and *what an agent must hold in mind* before writing a single line of rendering code.

The UI has one job: make the gap between lexical (text/grep) analysis and full compiler-semantic
analysis viscerally obvious to an engineer in under 60 seconds. Every element below exists to serve
that job or to make the tool feel alive during a live demo. Elements that don't serve one of those
two purposes should be cut.

---

## Rendering Environment Constraints

Before specifying elements, the constraints that govern all of them:

**Terminal is a line-based medium.** Unlike a browser, the terminal has no layout engine. Horizontal
composition (columns) must be constructed manually using fixed-width string padding. This is not a
limitation to work around — it is a fundamental constraint to design with.

**Kotter renders a single "live section" in-place.** State changes trigger a re-render of the active
section. Output that should persist (prior phases, scrolled history) must be explicitly flushed to
`aside`, which renders above the live section and does not re-render. The live section is always the
bottom portion of the screen.

**Terminal width is a runtime value, not a constant.** All column math must be derived from the
detected terminal width at session start. Hardcoding widths produces broken output on non-standard
terminals and on CI. This value must be queried once, stored, and threaded through every layout
calculation that depends on it.

**Kotter's `run` block is a coroutine scope.** This is an asset, not a footnote. Parallel animations
are expressed as concurrent coroutine launches — which is more natural in Kotlin than the
`setTimeout` chains used in the reference implementation. An agent should lean into this rather than
serializing everything.

## Element Specifications

### 1. Operation Rail

**What it is.** A horizontal row of labeled operation buttons rendered at the top of the screen.
Each button represents one supported `kast` operation (Rename, Call Hierarchy, Find References, etc.).
The active operation is visually distinguished. Inactive operations are dimmed. The rail is static
— it does not scroll or update during analysis.

**Why it's valuable.** The rail is the primary navigation surface and the first thing a demo
audience sees. It communicates immediately that this is a multi-capability tool, not a single
command. It also gives the person running the demo a clear, low-friction way to switch operations
mid-presentation without retyping commands. The demo's engagement depends on being able to pivot
between scenarios quickly.

**Kotter feasibility.** Fully feasible. The rail is a single `textLine()` composed of colored
text segments. Keyboard input (arrow keys or letter shortcuts) updates an `activeOp` variable,
which triggers a section re-render highlighting the new selection. No special primitives required.

**Agent planning notes.** The rail must be part of the persistent header — rendered in `aside`
once per selection change, not on every animation tick. If it lives in the live section, it
re-renders with every streaming line and creates flicker. Decide whether the rail is redrawn on
selection change (re-flush to aside) or kept in the live section at the cost of re-render cost.
The former is correct.

---

### 2. Query Bar

**What it is.** A single-line display showing the full `kast` CLI command being executed, with
syntax highlighting applied to its tokens: the binary name, subcommand, flag names, and flag
values each receive distinct colors. A blinking cursor appears at the end of the command while
analysis is running, then disappears on completion.

**Why it's valuable.** The query bar does double duty. First, it grounds the demo in reality —
the audience sees a real, executable command, not a magic black box. Second, the syntax coloring
signals that `kast` understands the structure of its own invocation. The cursor blink is a
behavioral signal that analysis is live, not pre-recorded. Both effects cost almost nothing to
implement and meaningfully raise the "this is real" sensation.

**Kotter feasibility.** Fully feasible. Token-splitting the command string and applying Kotter
color/style wrappers per token is trivial. The cursor blink is a coroutine loop toggling a
boolean, which triggers re-render of the cursor character.

**Agent planning notes.** The query bar should be rendered in the live section, not aside,
because the cursor must blink. This means it re-renders on every animation tick. Keep it at
the top of the live section with a clear visual separator below it so it reads as a fixed header
even as content streams below it.

---

### 3. Phase Bar

**What it is.** A horizontal sequence of named compiler stages — for example,
`RESOLVE → INDEX → PROPAGATE → APPLY` — rendered as labeled nodes connected by line segments.
Each stage transitions through three visual states: pending (dim, no prefix), active (highlighted,
play indicator prefix), and complete (accent color, checkmark prefix). Transitions happen at
defined points in the animation sequence, not on a timer.

**Why it's valuable.** Compiler analysis is opaque. Without the phase bar, the audience watches
text stream in with no mental model of what work is happening. The phase bar names the stages of
compiler work and shows progress through them, which serves two goals simultaneously: it makes
the tool feel principled (it has stages, it knows where it is), and it gives the audience
something to watch during gaps in streaming output. The transitions are pegged to real progress
points, not fake progress — which matters for credibility.

**Kotter feasibility.** Fully feasible. Three visual states per node, connected by separator
characters. Re-rendered whenever active phase index changes. Phase transitions are triggered by
the animation coroutine at the appropriate points in the sequence, not by timers.

**Agent planning notes.** Phase transitions must be driven by animation milestones, not wall-clock
delays. An agent implementing this should define explicit transition points in the scenario data
(e.g., "after stream1 completes, advance to phase 2; after branches complete, advance to phase 3")

### 4. Activity Indicator

**What it is.** A small pulsing dot rendered adjacent to the phase bar. While analysis is running,
it alternates between filled and hollow states on a fixed interval. When analysis completes, it
renders as a static filled dot in the accent color.

**Why it's valuable.** The dot is cheap to implement and disproportionately valuable during the
periods between streaming lines — particularly during the branch traversal, where the grid is
updating but column-by-column re-renders can feel stilted. The dot provides a continuous "alive"
signal without requiring meaningful content to be produced every frame.

**Kotter feasibility.** Fully feasible. A coroutine toggles a boolean on a ~500ms interval while
`running` is true. The dot character and its fill state respond to the boolean. Negligible render
cost.

**Agent planning notes.** This should be the last element implemented. It adds polish but has
zero impact on the core communication goal. If implementation time is constrained, skip it and
spend that time on the branch block or color system.

### 5. Semantic Color System

**What it is.** A defined mapping from semantic token types to specific colors, applied
consistently across all rendered output. Token types include: command text, compiler-confirmed
results, ambiguity warnings, unresolvable errors, resolved values, PSI/symbol node labels,
override chain entries, skipped/excluded items, property keys, property values, and structural
delimiters (brackets, arrows).

**Why it's valuable.** The color system is the primary mechanism by which a reader scans output
and extracts meaning without reading every word. Green means confirmed. Amber means flag for
review. Red means unresolvable. A dim color means structural scaffolding, not content. This
encoding must be consistent across all operations — if green means "compiler-confirmed" in Rename,
it must mean the same in Diagnostics. The moment the system is inconsistent, the audience stops
trusting the colors and starts reading every line, which defeats the purpose.

**Kotter feasibility.** Fully feasible. Kotter supports ANSI 256-color and truecolor via its
color API. The color system should be defined as a sealed object or companion with named constants
— not as inline hex strings scattered through rendering code.

**Agent planning notes.** Define the color system as a single source of truth before writing any
rendering code. Every element in the UI draws from it. An agent that hardcodes colors inline will
produce a system that is nearly impossible to audit for consistency. The color constants should be
named semantically (`Confirmed`, `Flagged`, `Unresolvable`, `NodeLabel`, etc.), not by hue
(`Green`, `Amber`). This is a naming discipline issue that compounds badly at scale.

---

### 6. Stream Block

**What it is.** A sequence of lines that appear one at a time in the terminal output, each after
a defined delay. Each line carries a semantic type that determines its rendering color and weight
per the color system. Lines with `type = sep` render as blank vertical space. All lines are
pre-authored in scenario data — the stream block is a playback engine, not a generator.

**Why it's valuable.** The streaming effect is the primary carrier of the "this is running for
real" sensation. Static output reads as documentation. Output that arrives line by line reads as
live analysis. The delays between lines do not need to simulate actual processing time — they need
to be fast enough to feel responsive and slow enough for a line to register before the next one
arrives. ~80–200ms per line is the target range.

**Kotter feasibility.** Fully feasible. Each line is a scheduled coroutine launch with a delay
offset. On each launch, the visible line list grows by one and the section re-renders. No
threading complexity.

**Agent planning notes.** Line delays should be defined in scenario data as offsets from the
block start time, not as sequential sleeps. This decouples the timing of individual lines from
each other and makes it possible to overlap stream phases with branch animation. An agent
implementing this as `delay(80)` in a loop will produce correct output but will block branch
animation from starting concurrently.

---

### 7. Parallel Branch Block

**What it is.** A fixed-width multi-column grid rendered during operations that traverse multiple
independent paths simultaneously — specifically Rename and Call Hierarchy. Each column represents
one traversal branch (e.g., one file being renamed, one call chain being walked). All columns
begin populating at the same time. Lines within each column appear sequentially with per-branch
timing. Each column has a header (the file or class name) in a distinct accent color, and a brief
summary line that appears after all branch lines are complete.

The grid is assembled programmatically: column width = `(terminalWidth - totalPadding) / branchCount`.
At each render tick, the full grid is rebuilt row by row: for row `i`, render `branch[k].lines[i]`
if it exists, or fill with spaces to the column width if that branch has not yet reached row `i`.
This produces independent per-column progression within a shared grid frame.

**Why it's valuable.** This is the highest-value element in the entire UI. No grep-based tool can
show this. The parallel branch view makes the claim "the compiler is simultaneously resolving
multiple independent paths" *visible* rather than asserted. It is the moment in a demo where the
audience stops passively watching and starts leaning forward. Every other element in the UI
supports this moment — the phase bar names the stage, the colors confirm success or flag risk,
the stream block sets up the context. The branch block is the payoff.

Its secondary value is precision: each branch shows only the call sites that resolve to the
target symbol. Lexical search would show 12 noise matches. The branch block shows 4 columns with
exactly the affected sites, annotated with call kind and risk flags.

**Kotter feasibility.** Feasible with explicit implementation attention. The key constraint is
that Kotter has no layout engine — column composition must be done in rendering code, not by the
framework. This requires: terminal width detection at startup, fixed column width calculation, a
per-render function that assembles the grid string row-by-row from current branch state, and
care to ensure lines longer than the column width are truncated or wrapped to avoid breaking
the grid.

The concurrent population of branches is natural in Kotter: each branch is a `launch` block in
the `run` coroutine scope, sequencing its own lines with `delay()`. The grid renderer reads
whatever each branch has produced so far on each re-render tick.

### 8. Output Region Management

**What it is.** A deliberate partitioning of terminal output into two regions: the *live section*
(Kotter's re-renderable section at the bottom of the screen), which contains the currently active
phase output and the branch block; and the *persistent region* (Kotter's `aside`), which contains
everything that has already completed and should not re-render. When a phase completes, its
output is flushed to `aside` and removed from the live section.

**Why it's valuable.** Without this partitioning, a naive implementation re-renders all output
on every state change — which in Kotter means clearing and redrawing the entire visible section
every ~100ms. This produces visible flicker and, more importantly, erases the audience's ability
to scroll back and read prior phase output. The partition preserves completed output as stable
terminal history while keeping the active section fast and clean.

**Kotter feasibility.** This is the idiomatic Kotter pattern. `aside {}` is specifically designed
for this use case. The agent should flush phase output to `aside` at phase transition points.

**Agent planning notes.** The boundary between `aside` and the live section is the primary
architectural decision in the rendering layer. An agent should establish this boundary before
implementing any individual element. A reasonable partition: the operation rail and query bar
live in the live section (they need to update). Completed stream blocks move to `aside` when
their phase ends. The branch block lives in the live section until complete, then flushes to
`aside`. The phase bar and activity indicator always live in the live section.

---

### 9. Animation Timing Model

**What it is.** A scenario data structure that encodes all line delays as absolute time offsets
from a block start time, not as relative delays between adjacent lines. Each scenario defines
stream phases and branch phases. Stream lines carry a `t` offset in milliseconds. Branch lines
carry a `t` offset from branch start. Block start times are derived from prior block completion
times at runtime — the timing model is relative within blocks, absolute across the animation
sequence.

**Why it's valuable.** The timing model is what enables parallel animation. If delays are
sequential (sleep N ms, emit line, sleep N ms, emit line), branches cannot run concurrently —
each line in branch A blocks branch B. Offset-based timing allows all branch coroutines to
be launched simultaneously, each scheduling its own lines independently. This is what produces
the parallel traversal effect.

**Kotter feasibility.** Offset-based timing with `launch` + `delay(offset)` per line is idiomatic
Kotlin coroutine usage and maps cleanly to Kotter's `run` block.

**Agent planning notes.** Scenario data should encode timing offsets, not business logic. The
rendering layer reads offsets and applies them; it does not compute timing. This separation means
timing can be tuned (slow down for a live demo audience, speed up for a CI recording) by changing
scenario data, not rendering code. An agent that bakes timing math into the renderer has created
a maintenance problem.

---

### 10. Keyboard Interaction Model

**What it is.** A minimal keyboard interface supporting: operation switching via letter shortcuts
(one key per operation), replay of the current scenario, and quit. No mouse support. No free-text
input in the demo mode. The active operation key is displayed inline on each operation button in
the rail.

**Why it's valuable.** The demo context is a live audience presentation. Free-text input creates
failure modes (typos, unrecognized symbols, slow typing). Pre-authored scenarios with keyboard
shortcuts give the presenter reliable control without requiring them to type accurately under
pressure. The interaction model is deliberately narrow: pick an operation, watch it run, pick
another. That is the entire required interaction surface for the demo use case.

**Kotter feasibility.** Fully feasible. Kotter's `onKeyPressed` handler dispatches on key values.
Letter shortcuts map to operation IDs. Any key press while an animation is running should be
queued, not dropped — switching operation mid-animation should cancel the current run cleanly
and start the new one.

**Agent planning notes.** Animation cancellation is the edge case to handle correctly. When the
user switches operations, all pending coroutine launches for the current scenario must be
cancelled before the new scenario starts. Kotter's structured concurrency makes this manageable:
the current scenario's coroutines should be launched inside a `CoroutineScope` that is cancelled
on operation switch. An agent that does not cancel pending launches will produce interleaved
output from two scenarios on rapid switching.

For planning and phasing purposes:

**Ship-blocking.** The branch block, semantic color system, stream block, and output region

**High value, low risk.** The phase bar, query bar, and operation rail are fast to implement and
significantly raise perceived quality. Build them early.

**Polish, defer if needed.** The activity indicator and keyboard cancellation robustness can be
deferred to a second pass without affecting the demo goal.

be evaluated against that goal, not against general software quality criteria.

    /**
     * Choose the [Symbol] that best matches a user filter. Prefers an exact
     * `fqName` match so callers can disambiguate overloaded simple names by
     * passing a fully-qualified class name; falls back through suffix and
     * substring matches in that order.
     */
    private fun pickBestMatch(symbols: List<Symbol>, filter: String): Symbol? {
        val exact = symbols.firstOrNull { it.fqName == filter }
        if (exact != null) return exact
        val suffix = symbols.firstOrNull { it.fqName.endsWith(".$filter") }
        if (suffix != null) return suffix
        val simple = symbols.firstOrNull { it.fqName.substringAfterLast('.') == filter }

        line: String,

        while (index >= 0) {

            val after = line.getOrNull(index + symbolName.length)
            if (before.isIdentifierBoundaryParticipant() || after.isIdentifierBoundaryParticipant()) {
                return true

            index = line.indexOf(symbolName, startIndex = index + 1)

    private fun StringBuilder.renderPanel(panel: DemoScene.Panel) {

        val contentWidth = inner - 2
        val border = styleAnsi("1;36", "─".repeat(inner))
        val topRule = "${styleAnsi("1;36", "┌")}$border${styleAnsi("1;36", "┐")}"
        val bottomRule = "${styleAnsi("1;36", "└")}$border${styleAnsi("1;36", "┘")}"
        val separator = "${styleAnsi("1;36", "├")}$border${styleAnsi("1;36", "┤")}"
        appendLine(topRule)
        appendLine(renderPanelRow(styleAnsi("1;37", truncate(panel.title, contentWidth)), truncate(panel.title, contentWidth).length, contentWidth))

        panel.lines.forEach { line ->

                // Pre-styled lines own their own ANSI / truncation. We still
                // pad them out to contentWidth using the plain text length.
                val rawLength = line.text.length.coerceAtMost(contentWidth)
                val rendered = if (line.text.length > contentWidth) {

                    emphasise(line.emphasis, truncate(line.text, contentWidth))

                    line.prerendered

                return@forEach

            if (segments.isEmpty()) {
                appendLine(renderPanelRow("", 0, contentWidth))
            } else {
                segments.forEach { segment ->
                    appendLine(renderPanelRow(emphasise(line.emphasis, segment), segment.length, contentWidth))
                }
            }
        }
        appendLine(bottomRule)
    }

    private fun renderPanelRow(rendered: String, rawLength: Int, contentWidth: Int): String {
        val pad = padTo(rawLength, contentWidth)
        return "${styleAnsi("1;36", "│ ")}$rendered$pad${styleAnsi("1;36", " │")}"

    private fun StringBuilder.renderComparisonTable(table: DemoScene.ComparisonTable) {

        val natLeft = max(table.header.second.length, table.rows.maxOfOrNull { it.second.length } ?: 0)

        return leftShare.coerceAtMost(leftNatural.coerceAtLeast(MIN_CELL)) to

        LineEmphasis.DIM -> styleAnsi("2", text)

        LineEmphasis.SUCCESS -> styleAnsi("1;32", text)

            while (j < text.length && text[j].isWhitespace() == isSpace) j += 1
            tokens += text.substring(i, j)
            i = j
        }
        return tokens

/**
 * Declarative description of a `kast demo` scene. A [DemoScript] is a flat,
 * ordered list of [DemoScene] nodes built through [demoScript]. The renderer
 * walks the tree once and produces an ANSI/markdown string; the DSL lets the
 * command flow stay declarative and lets tests assert on the tree before any

    val scenes: List<DemoScene>,

/**
 * A single scene in the demo. Sealed so the renderer exhausts every case and
 * new scene kinds cannot silently slip through.
 */
internal sealed interface DemoScene {
    /** Top or closing banner rendered as a boxed panel. */
    data class Panel(
        val title: String,
        val lines: List<PanelLine>,
    ) : DemoScene

    /** Section heading, e.g. `── Act 1 · text search baseline ──`. */
    data class SectionHeading(
        val title: String,
    ) : DemoScene

    /** One-line step progress indicator, e.g. `› Warming workspace daemon...`. */
    data class StepProgress(
        val message: String,
    ) : DemoScene

    /** Step outcome with elapsed wall-clock time. */
    data class StepOutcome(
        val message: String,
        val outcome: StepResult,
        val elapsed: Duration?,
    ) : DemoScene

    /** Indented body of a rendered step (e.g. resolved symbol, ref list). */
    data class StepBody(
        val lines: List<BodyLine>,
    ) : DemoScene

    /** Side-by-side comparison table. */
    data class ComparisonTable(
        val header: Triple<String, String, String>,
        val rows: List<Triple<String, String, String>>,

    internal fun renderCallTree(workspaceRoot: Path, root: CallNode): List<String> {
        val lines = mutableListOf<String>()
        val remaining = intArrayOf(CALL_TREE_LIMIT)

        fun walk(node: CallNode, depth: Int) {
            if (remaining[0] <= 0) return
            remaining[0] -= 1
            val indent = "  ".repeat(depth.coerceAtLeast(0))
            val symbol = node.symbol
            val prefix = if (depth > 0) "├─ " else ""
            val location = Paths.locationLine(workspaceRoot, symbol.location)
            lines += "$indent$prefix${symbol.fqName.substringAfterLast('.')} (${symbol.kind})  $location"
            node.children.forEach { child -> walk(child, depth + 1) }
        }

        walk(root, depth = 0)
        return lines
    }
